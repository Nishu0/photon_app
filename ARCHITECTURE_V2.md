# Kodama Architecture v2

v1 is a single tool-use loop against one LLM with every tool registered at once. That works until the tool list grows past ~15 entries and the model picks wrong tools, and until memory grows past a few hundred rows and the prompt stops being useful. v2 fixes both and adds real guardrails.

## What changes from v1

| | v1 | v2 |
| --- | --- | --- |
| Agent shape | single loop, all tools visible | parent dispatcher + specialized sub-agents, scoped tools |
| SDK | AI SDK v6 + OpenRouter | `@anthropic-ai/claude-agent-sdk` |
| DB | `bun:sqlite` | Convex (reactive) |
| Memory | flat rows, never forgets | 7 segments × 3 buckets × decay |
| Curation | manual | nightly adversarial cleanup (consolidator → adversary → judge) |
| Cost | unbounded | per-agent $ cap, recursion guard, turn cap |

## Shape

```
iMessage ─► parent agent ─► dispatch(agent, instructions)
                             │
                             ├─► email      (gmail tools + memory)
                             ├─► twitter    (twitterapi tools + memory)
                             ├─► weather    (weather api + memory)
                             ├─► youtube    (youtube tools + memory)
                             ├─► journal    (log/recap + memory)
                             ├─► tasks      (add/complete/snooze + memory)
                             └─► memory     (search / save / recall)
```

Every sub-agent:
- starts a fresh `query()` from `@anthropic-ai/claude-agent-sdk` with its own system prompt and tool whitelist
- has a per-run spend cap ($40 default)
- **cannot** spawn further sub-agents (recursion guard)
- reads/writes memory through a thin adapter — never touches the db directly

## Pieces

### 1. Parent agent (dispatcher)

One tool: `dispatch_to_agent(name, instructions)`. That's it.

```ts
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const event of query({
  prompt: userMessage,
  options: {
    systemPrompt: parentPrompt,
    allowedTools: ["dispatch_to_agent"],
    maxTurns: 4,
  }
})) {
  // stream parent decisions
}
```

The parent owns: intent routing, reply-style choice (tapback vs text), and stitching sub-agent outputs into the final iMessage reply. No domain logic.

### 2. Sub-agents

Each is `query()` with its own prompt + tool list + memory slice.

| Agent | Tools |
| --- | --- |
| `email` | `gmail_list_unread`, `gmail_search`, `gmail_read`, `gmail_create_draft`, `gmail_send` (all pass through v1's redaction policy) |
| `twitter` | `x_watch_user`, `x_unwatch_user`, `x_list_watches`, `x_recent_for_user`, `x_digest_now` |
| `weather` | `weather_current`, `weather_forecast` |
| `youtube` | `yt_search`, `yt_channel_info`, `yt_transcript` |
| `journal` | `log_journal`, `recap` |
| `tasks` | `add_task`, `complete_task`, `snooze_task`, `drop_task`, `schedule_reminder` |
| `memory` | `memory_save`, `memory_recall`, `memory_forget` |

Every sub-agent also has read access to memory_recall (implicit system prompt injection, not a tool call).

### 3. Memory system

**Three buckets** with different lifetimes:

| Bucket | Lifetime | Example |
| --- | --- | --- |
| `short_term` | decays fast | "researching japan", "going to SF in 1 week" |
| `long_term` | decays slow | "chose sendblue for iMessage API", "cousin in military" |
| `permanent` | never decays | "YouTube channel is @raroque", "was econ major at Vanderbilt" |

**Seven segments**, each with a default importance and decay behavior:

| Segment | Default | Decay |
| --- | --- | --- |
| Identity | 0.90 | promotes to permanent quickly |
| Correction | 0.80 | long-term, slow decay |
| Preference | 0.70 | long-term, adaptive |
| Relationship | 0.65 | long-term, adaptive |
| Knowledge | 0.60 | long-term, adaptive |
| Behavioral | 0.55 | long-term, generated by reflection |
| Context | 0.40 | short-term, 8% decay/day |

Decay is computed on read:

```
effective_importance = stored_importance * exp(-decay_rate * days_since_last_access)
```

A memory below a floor (e.g. 0.1) after decay is a prune candidate for the nightly pass. Accessing a memory refreshes its `lastAccessedAt` and bumps `accessCount`. Repeated access promotes across buckets.

### 4. Nightly cleanup (adversarial, 03:00 local)

```
All memories ─► Consolidator (Sonnet) ─► Adversary (Sonnet) ─► contested?
                                                                  ├── no → Auto-approve ─► apply
                                                                  └── yes → Debate ≤2 rounds → Judge (Opus) → apply
```

- **Consolidator** — for each memory touched in the last 24h, proposes `promote | merge | prune | keep`.
- **Adversary** — pushes back on every proposal with a "why this should stay" / "why this shouldn't change" argument.
- **Debate** — up to 2 rounds between them.
- **Judge (Opus)** — tiebreaker when they can't agree. Approves, rejects, or modifies.
- **Apply** — one Convex transaction, audit row per action in `memoryEvents`.

### 5. Data layer (Convex)

```ts
// convex/schema.ts
export default defineSchema({
  conversations: defineTable({
    userId: v.string(),
    visibleId: v.string(),
    title: v.optional(v.string()),
    sessionId: v.optional(v.string()), // claude-agent-sdk resume
    usage: v.optional(v.object({ inputTokens: v.number(), outputTokens: v.number(), cost: v.number(), totalDurationMs: v.number() })),
    createdAt: v.number(),
    updatedAt: v.number(),
  }).index("by_user", ["userId"]).index("by_visibleId", ["visibleId"]),

  messages: defineTable({
    conversationId: v.id("conversations"),
    agentId: v.optional(v.string()),
    role: v.union(v.literal("user"), v.literal("assistant"), v.literal("system")),
    content: v.string(),
    createdAt: v.number(),
  }).index("by_conversation", ["conversationId"]),

  memories: defineTable({
    userId: v.string(),
    segment: v.union(
      v.literal("identity"), v.literal("preference"), v.literal("correction"),
      v.literal("relationship"), v.literal("knowledge"), v.literal("behavioral"), v.literal("context"),
    ),
    bucket: v.union(v.literal("short_term"), v.literal("long_term"), v.literal("permanent")),
    content: v.string(),
    importance: v.number(),
    decayRate: v.number(),
    createdAt: v.number(),
    lastAccessedAt: v.number(),
    accessCount: v.number(),
  }).index("by_user", ["userId"]).index("by_user_segment", ["userId", "segment"]),

  memoryEvents: defineTable({
    memoryId: v.id("memories"),
    action: v.union(v.literal("created"), v.literal("promoted"), v.literal("merged"), v.literal("pruned"), v.literal("accessed")),
    reason: v.optional(v.string()),
    actor: v.optional(v.string()), // "consolidator" | "judge" | "user" | "agent"
    at: v.number(),
  }).index("by_memory", ["memoryId"]),

  spendLedger: defineTable({
    runId: v.string(),
    agentName: v.string(),
    inputTokens: v.number(),
    outputTokens: v.number(),
    usd: v.number(),
    at: v.number(),
  }).index("by_run", ["runId"]),

  sessionState: defineTable({
    userId: v.string(),
    sessionId: v.string(),
    agentName: v.string(),
    snapshot: v.any(),
    updatedAt: v.number(),
  }).index("by_user_agent", ["userId", "agentName"]),
});
```

### 6. Guardrails

- **Spend cap** — per-agent USD cap enforced by intercepting the SDK stream and aborting when `spendLedger` sum for the run exceeds the cap. Default $40. Parent cap separate, $5.
- **Recursion guard** — sub-agents have `dispatch_to_agent` **removed** from `allowedTools`. Enforced at the agent factory layer.
- **Turn cap** — `maxTurns: 4` default, per-agent override in the registry.
- **Panic abort** — if an agent fails twice in a row, the parent short-circuits to "something's off, try again later" without further spend.

## Migration plan

v1 keeps running. v2 lives under `src/v2/`. An env flag `KODAMA_V2=1` flips the daemon's message pump to v2. Once v2 is stable, `src/` is deleted and `src/v2/` is renamed.

## Open questions

1. **Convex vs local-first.** v1's pitch was "your data on your Mac." Convex breaks that. Options: (a) accept the tradeoff, (b) mirror every Convex write to a local sqlite so offline still works, (c) stay on sqlite with reactive wrappers. Need to decide before shipping.
2. **Model routing.** Claude Agent SDK is Anthropic-only. OK for now; OpenRouter re-integration is possible later by swapping the SDK layer if needed.
3. **Memory classification cost.** Every `memory_save` calling a Sonnet classifier is ~100 tokens per save. Cheap, but adds up. Batched nightly classification is an alternative.
